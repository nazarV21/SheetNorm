# SheetNorm

**Превращает сложные Excel-отчёты в готовые данные для аналитики.**

SheetNorm — локальная система с AI-помощником для автоматической подготовки сложных Excel-отчётов к загрузке в 1С, Power BI, SQL, BI-системы и корпоративные хранилища данных.

AI-помощник не заменяет проверку человеком: он анализирует структуру и формирует проект правила. Пользователь проверяет предпросмотр, уточняет инструкцию и сохраняет результат в повторяемом шаблоне.

## Какую проблему решает

Корпоративные Excel-файлы часто содержат служебные строки, объединённые ячейки, многоуровневые заголовки, периоды в колонках и итоги внутри таблицы. Такие отчёты удобны для чтения, но требуют ручной очистки перед загрузкой в информационные системы.

SheetNorm сокращает эту ручную работу:

- распознаёт структуру исходного файла;
- преобразует кросс-таблицы в плоский формат;
- удаляет служебные и итоговые строки по проверяемому правилу;
- сохраняет утверждённую логику как шаблон;
- формирует готовый `.xlsx` и отчёт контроля результата.

## Кому полезен

- аналитикам и BI-командам;
- финансовым и производственным подразделениям;
- специалистам по 1С и корпоративным хранилищам;
- командам, регулярно получающим отчёты от филиалов и подрядчиков;
- компаниям, которым важна локальная обработка конфиденциальных файлов.

## Ключевые возможности

- загрузка `.xlsx` и `.xls` через Flask-интерфейс или API;
- работа без локальной LLM через детерминированные эвристики;
- опциональная локальная GGUF-модель без облачных AI API;
- пошаговый AI-помощник с исходным и итоговым предпросмотром;
- библиотека шаблонов с поиском, фильтрами, копированием и JSON import/export;
- память обучающих пар `source → target` и исправлений инструкций;
- контроль строк, колонок, пустых ячеек, операций и предупреждений;
- Excel-экспорт с автофильтром, закреплением заголовка и листом `processing_summary`;
- структурированные API-ошибки и ограничение размера загрузки;
- Docker-конфигурация для внутреннего развёртывания.

## Запуск без LLM

Это основной безопасный режим по умолчанию:

```text
AI_BACKEND=fallback
```

Анализ структуры, формализованные правила, предпросмотр, шаблоны и экспорт работают без файла модели.

## Локальная LLM

LLM работает только внутри контура приложения. Excel-файлы не отправляются во внешние AI API.

```powershell
python -m pip install -r requirements-ai.txt
powershell -ExecutionPolicy Bypass -File scripts/download_model.ps1
```

Настройте `.env`:

```text
AI_BACKEND=llama_cpp
AI_MODEL_PATH=models/qwen2.5-3b-instruct-q4_k_m.gguf
```

GGUF-модель не включается в Git или Docker-образ.

## Основной пользовательский сценарий

1. Загрузить Excel в AI-помощник.
2. Проверить автоматически определённые заголовки и начало данных.
3. Описать результат простыми словами.
4. Проверить уточнённую инструкцию и внутреннее правило.
5. Сравнить исходный и итоговый предпросмотр.
6. Сохранить проверенную логику как шаблон.
7. Скачать результат и применять шаблон к следующим файлам.

Пример инструкции:

```text
Заголовки на строке 4, данные начинаются со строки 5. Удали строки Итого и Всего.
Первые три столбца оставь идентификаторами, месяцы перенеси в колонку Период.
```

## Демо-сценарий для показа заказчику

1. Запустите приложение в режиме `AI_BACKEND=fallback`.
2. Откройте Dashboard и покажите локальный контур обработки.
3. Перейдите в AI-помощник и загрузите типовой сложный файл.
4. Покажите определённые строки заголовков и предпросмотр.
5. Уточните инструкцию одной фразой и обновите результат.
6. Сохраните шаблон и скачайте `.xlsx`.
7. Откройте `processing_summary` в итоговом файле.
8. Повторно обработайте похожий файл через `/convert`.

Демонстрационные шаблоны восстанавливаются командой:

```bash
python scripts/seed_demo_rules.py
```

## Что показать за 3 минуты

- Dashboard: назначение, локальная обработка, количество шаблонов.
- AI-помощник: анализ структуры и инструкция на естественном языке.
- Контроль результата: строки, колонки, пустые ячейки и предупреждения.
- Шаблоны: повторное использование утверждённой логики.
- Итоговый Excel: `Результат` и `processing_summary`.

## Задачи обработки и статусы

Каждая загрузка создаёт `ProcessingJob` в `jobs.json`. Поддерживаются статусы `created`, `queued`, `processing`, `success`, `failed`, `cancelled`. Страница `/jobs` показывает список задач, а `/jobs/<job_id>` — полный отчёт, ошибки, предупреждения, метрики и предпросмотр результата.

`jobs.json` — новая pilot-ready сущность. `history.json` временно остаётся legacy-журналом для обратной совместимости. Перед многопользовательским внедрением оба механизма должны быть перенесены в PostgreSQL.

## Отчёт качества

Каждая успешная обработка формирует обязательный `quality_report`: строки и колонки до/после, пустые ячейки, определённый тип таблицы, операции, предупреждения и статус `success`, `warning` или `error`. Отчёт доступен на странице результата, в ProcessingJob и на листе `processing_summary` итогового Excel.

Если метрику невозможно рассчитать, обработка не падает: поле получает `null`, а отчёт — предупреждение.

## Пилотное внедрение

Текущая версия готова для контролируемого пилота: локальный запуск, проверка шаблонов на реальных файлах, измерение качества и сбор требований. На пилоте определяются фактическая нагрузка, роли, retention файлов и интеграции.

## SheetNorm Team и SheetNorm Enterprise

- **Team**: целевая конфигурация для отдела, ориентир 10–30 пользователей и сотни файлов в месяц. Требует PostgreSQL, очереди задач и базовых ролей.
- **Enterprise**: целевая конфигурация для нескольких подразделений, ориентир 30–100+ пользователей и тысячи файлов в месяц. Требует PostgreSQL, Redis/RQ или Celery, workers, файлового хранилища, аудита и мониторинга.

Промышленная конфигурация подбирается после пилота и тестирования на файлах компании. Цифры нагрузки не являются гарантией до нагрузочного тестирования. Подробности: [`docs/commercial_packages.md`](docs/commercial_packages.md).

## Production architecture roadmap

Целевая схема: Web App + PostgreSQL + Redis/RQ + Workers + File Storage. Локальная LLM остаётся опциональной и помогает создавать правила; массовая обработка должна выполняться по сохранённым детерминированным шаблонам. Подробности: [`docs/production_architecture.md`](docs/production_architecture.md).

## Ограничения текущей версии

- эвристики не гарантируют идеальное определение любой нестандартной формы;
- `.xls` требует установленного `xlrd`;
- JSON-файлы подходят для MVP и пилота, но для многопользовательской production-среды нужна БД;
- текущая аутентификация и ролевая модель не реализованы;
- оценка уверенности является практическим индикатором, а не статистической гарантией качества;
- финальный результат должен проходить бизнес-проверку владельцем данных.
