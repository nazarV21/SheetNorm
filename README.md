# SheetNorm

**Превращает сложные Excel-отчёты в готовые данные для аналитики.**

SheetNorm — локальная система с AI-помощником для автоматической подготовки сложных Excel-отчётов к загрузке в 1С, Power BI, SQL, BI-системы и корпоративные хранилища данных.

AI-помощник не заменяет проверку человеком: он анализирует структуру и формирует проект правила. Пользователь проверяет предпросмотр, уточняет инструкцию и сохраняет результат в повторяемом шаблоне.

## Какую проблему решает

Корпоративные Excel-файлы часто содержат служебные строки, объединённые ячейки, многоуровневые заголовки, периоды в колонках и итоги внутри таблицы. Такие отчёты удобны для чтения, но требуют ручной очистки перед загрузкой в информационные системы.

SheetNorm сокращает эту ручную работу:

- распознаёт структуру исходного файла;
- преобразует кросс-таблицы в плоский формат;
- удаляет служебные и итоговые строки по проверяемому правилу;
- сохраняет утверждённую логику как шаблон;
- формирует готовый `.xlsx` и отчёт контроля результата.

### Автоматический выбор локальной модели

Если в `models/` лежит несколько GGUF, SheetNorm может сам выбрать модель под конфигурацию компьютера. В `.env` уже включены безопасные настройки:

```env
AI_MODEL_SELECTION_MODE=auto
AI_AUTO_SELECT_ON_STARTUP=true
AI_AUTO_ACTIVATE=true
AI_AUTO_TEST=true
AI_RESELECT_IF_UNAVAILABLE=true
AI_MIN_FREE_RAM_GB=2
AI_MAX_RAM_USAGE_RATIO=0.72
```

Модель назначается при старте, но не занимает RAM до первого AI-запроса. Статус «модель для AI назначена, в RAM не загружена» является нормальным. Подробности: `docs/model_management.md`.

## Локальная LLM и выбор модели

LLM работает внутри контура приложения. Excel-файлы не отправляются во внешние AI API.

Установите поддержку llama.cpp:

```powershell
python -m pip install -r requirements-ai.txt
```

Поместите одну или несколько моделей `*.gguf` в папку `models/`, запустите приложение и откройте:

```text
http://127.0.0.1:5000/settings
```

На странице настроек можно:

- обновить список файлов из `models`;
- сравнить размер моделей с доступной RAM;
- выбрать экономичный, сбалансированный или производительный профиль;
- протестировать загрузку модели;
- активировать модель только после успешного теста;
- отключить LLM и вернуться к fallback-режиму.

Выбор хранится в PostgreSQL/SQLite, а в JSON-режиме — в `ai_settings.json`. Интерфейс не изменяет `.env`.

Начальная конфигурация через `.env` остаётся доступной:

```text
AI_BACKEND=fallback
AI_MODELS_DIR=models
AI_MODEL_PATH=
AI_DEFAULT_PROFILE=balanced
```

GGUF-файлы не включаются в Git или Docker-образ. Подробности: `docs/model_management.md`.

## Задачи обработки и статусы

Каждая загрузка создаёт `ProcessingJob`. В режиме базы данных задачи сохраняются в PostgreSQL, а JSON остаётся fallback для тестов и legacy-импорта. Поддерживаются статусы `created`, `queued`, `processing`, `requires_review`, `success`, `failed`, `cancelled`.

После отправки формы конвертация выполняется в фоне. Пользователь может перейти в другой раздел или закрыть вкладку — обработка продолжится на сервере. Страница `/jobs` показывает список, прогресс и этап выполнения, а `/jobs/<job_id>` — подробный отчёт. Активную задачу можно остановить из списка или карточки задачи. Отмена кооперативная: текущая операция pandas может завершить шаг, но отменённый результат не сохраняется как успешный.

Режимы выполнения:

- `ASYNC_MODE=thread` — локальный фоновой executor без Redis. Подходит для разработки и одиночного пилота; задачи не переживают перезапуск процесса приложения.
- `ASYNC_MODE=rq` — Redis/RQ worker. Рекомендуется для Docker, Team и длительных задач.
- `ASYNC_MODE=sync` — синхронный режим только для тестов и диагностики.

Для локального запуска без Redis добавьте в `.env`:

```env
ASYNC_MODE=thread
LOCAL_WORKER_THREADS=2
```


## Приоритет текущей инструкции

По умолчанию `AI_USE_LEARNED_HINTS=false`: сохранённые исправления не добавляются к новой пользовательской инструкции и не могут незаметно перенести старые фильтры, колонки или формат результата. Даже при включённой опции история передаётся модели только как структурная подсказка без старых текстов инструкций.

Для иерархических файлов поддерживается сценарий:

```text
разделить по филиалам на разные листы и сделать итог для каждого
```

Строки вида `Филиал Север` используются как разделители блоков, название филиала распространяется на строки данных, для каждого филиала создаётся отдельный лист, а внизу листа добавляется собственный итог по числовым колонкам.

## Отчёт качества

Каждая успешная обработка формирует обязательный `quality_report`: строки и колонки до/после, пустые ячейки, определённый тип таблицы, операции, предупреждения и статус `success`, `warning` или `error`. Отчёт доступен на странице задачи и в ProcessingJob, но не добавляется отдельным техническим листом в пользовательский Excel.

Если метрику невозможно рассчитать, обработка не падает: поле получает `null`, а отчёт — предупреждение.

## Пилотное внедрение

Текущая версия готова для контролируемого пилота: локальный запуск, проверка шаблонов на реальных файлах, измерение качества и сбор требований. На пилоте определяются фактическая нагрузка, роли, retention файлов и интеграции.

## SheetNorm Team и SheetNorm Enterprise

- **Team**: целевая конфигурация для отдела, ориентир 10–30 пользователей и сотни файлов в месяц. Требует PostgreSQL, очереди задач и базовых ролей.
- **Enterprise**: целевая конфигурация для нескольких подразделений, ориентир 30–100+ пользователей и тысячи файлов в месяц. Требует PostgreSQL, Redis/RQ или Celery, workers, файлового хранилища, аудита и мониторинга.

Промышленная конфигурация подбирается после пилота и тестирования на файлах компании. Цифры нагрузки не являются гарантией до нагрузочного тестирования. Подробности: [`docs/commercial_packages.md`](docs/commercial_packages.md).

## Production architecture roadmap

Целевая схема: Web App + PostgreSQL + Redis/RQ + Workers + File Storage. Локальная LLM остаётся опциональной и помогает создавать правила; массовая обработка должна выполняться по сохранённым детерминированным шаблонам. Подробности: [`docs/production_architecture.md`](docs/production_architecture.md).

## Ограничения текущей версии

- эвристики не гарантируют идеальное определение любой нестандартной формы;
- `.xls` требует установленного `xlrd`;
- JSON-файлы подходят для MVP и пилота, но для многопользовательской production-среды нужна БД;
- текущая аутентификация и ролевая модель не реализованы;
- оценка уверенности является практическим индикатором, а не статистической гарантией качества;
- финальный результат должен проходить бизнес-проверку владельцем данных.

## Новый порядок работы с AI-помощником

AI-помощник работает в два шага.

1. Загрузите Excel и сначала проверьте исходную таблицу: лист, строки заголовков, начало данных, служебные строки и строки итогов.
2. Если разметка неверная, напишите уточнение: `заголовки на строке 4, данные со строки 5`, `итоги удалить`, `не разворачивать в длинный формат`.
3. После проверки исходной таблицы опишите результат: какие колонки оставить, что переименовать, какие фильтры, итоги или melt нужны.
4. Только после этого система переформулирует задачу, соберёт внутреннее правило и построит итоговый preview.
5. Готовый Excel создаётся только после успешного preview.

Если поле инструкции пустое, приложение не строит итоговый preview и не применяет догадки как готовое правило.

## Пакетная обработка файлов и папок

На странице `/convert` доступны три сценария:

- один Excel-файл;
- несколько файлов за одну загрузку;
- папка с Excel-файлами.

Каждый файл создаёт отдельную фоновую задачу. Для группы файлов SheetNorm создаёт страницу пакета со статусами, общей остановкой и скачиванием готовых результатов одним ZIP.

В Chrome и Edge доступен режим «Выбрать папку и сохранить результаты рядом». Браузер запрашивает разрешение на чтение и запись, после чего готовые файлы сохраняются в подпапку `SheetNorm_results` выбранной папки. Из-за ограничений безопасности браузера этот режим требует явного выбора папки пользователем. В остальных браузерах используйте обычную загрузку папки и ZIP.

- В AI-помощнике файл отображается сразу после выбора, до запуска основного анализа.
- Листы показываются вкладками в интерфейсе, аналогично Excel.
- После загрузки система формирует подсказки по инструкции с учётом листов, строк заголовков, итогов, объединённых ячеек и структуры филиалов.
- В карточке завершённой задачи доступен предпросмотр всех итоговых листов.
- Технический лист `processing_summary` больше не добавляется в пользовательский Excel; отчёт качества хранится в карточке задачи.

## Универсальное разделение по листам

AI-помощник привязывает короткие инструкции к реальным колонкам загруженного файла. Например, команды `разделить по складам`, `каждый филиал на отдельный лист` и `создать лист для каждого менеджера` используют одну операцию группировки, а не отдельную жёстко заданную логику для каждой предметной области.

Перед выполнением система:

- определяет строку заголовков и начало данных;
- сопоставляет словоформу из инструкции с существующей колонкой;
- обнаруживает колонки с иерархическими пустыми значениями и при необходимости заполняет их сверху вниз;
- учитывает отрицания вроде `не добавлять итоги` и `не создавать сводный лист`;
- формирует много-листовый preview до финальной обработки.

Если запрос неоднозначен, например в файле одновременно есть `Склад отправления` и `Склад назначения`, правило не выполняется молча: пользователь получает просьбу уточнить колонку.

## Рабочее приложение и фоновые сессии

Внутренний интерфейс SheetNorm содержит только рабочие разделы: конвертацию, AI-помощник, шаблоны, задачи, обучающие примеры и настройки. Корневой URL открывает конвертацию. Маркетинговые страницы перенесены на сайт FormaOps.

AI-помощник создаёт сохраняемую рабочую сессию сразу после выбора файла. В разделе «Задачи» видны исходный промпт, текущий этап, выбранный лист, режим обработки, исходный и итоговый preview. Кнопка «Продолжить» возвращает пользователя к последнему сохранённому шагу.

Для локального запуска без Redis:

```env
ASYNC_MODE=thread
LOCAL_WORKER_THREADS=1
AI_MEMORY_MODE=economy
AI_IDLE_UNLOAD_SECONDS=300
SCRIPT_EXECUTION_ENABLED=false
```

Thread-режим продолжает работу после перехода между страницами или закрытия вкладки, но не переживает остановку серверного процесса. Для серверной установки используйте PostgreSQL + Redis/RQ. Локальная GGUF-модель загружается только при реальном AI-запросе и в экономичном режиме автоматически выгружается после простоя.

## Обновление существующей локальной установки

После замены исходного кода обязательно обновите схему базы данных:

```powershell
flask --app main db upgrade
flask --app main db-schema-status
```

Если локальная SQLite-база была создана промежуточной версией и Alembic уже считает старые миграции применёнными, в проекте есть идемпотентная ремонтная миграция `0005_schema_repair`. Для ручной диагностики доступны команды:

```powershell
flask --app main repair-local-db
flask --app main db upgrade
flask --app main db-schema-status
```

Во время выполнения `flask db ...` приложение не запускает автоматический подбор AI-модели и не обращается к новым колонкам до завершения миграции.

## Хранение истории задач и файлов

PostgreSQL или SQLite хранит метаданные задачи, промпт, статусы, этапы, пути к файлам и отчёт качества. Сами Excel-файлы хранятся в каталогах `input/` и `output/`, а не внутри BLOB-полей базы данных. Поэтому для сохранения полной истории нужно резервировать одновременно:

- базу данных;
- каталог `input/` с исходными файлами;
- каталог `output/` с итоговыми файлами;
- каталог `storage/`, если он используется выбранным storage backend.

В карточке выполняющейся задачи итоговый предпросмотр не показывается до появления реального выходного файла. После успешного завершения отображается именно тот Excel, который доступен по кнопке скачивания. Кнопка «Изменить и обработать заново» создаёт новый черновик AI-помощника и не перезаписывает историческую задачу.
