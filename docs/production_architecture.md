# Production architecture roadmap

## Текущая pilot-ready архитектура

SheetNorm работает как Flask-приложение с Jinja templates. Excel обрабатывается локально через pandas и openpyxl. Шаблоны, ProcessingJob, история и обучающие примеры хранятся в JSON. Конвертация выполняется синхронно внутри web-процесса.

Этот вариант прост для установки, демонстрации и пилота: не требует внешних сервисов и продолжает работать без локальной LLM. JSON при этом не обеспечивает транзакции, конкурентную запись, эффективную выборку и надёжный аудит для многопользовательской эксплуатации.

## Целевая схема

```text
Users / API clients
        |
     Web App
        |
  PostgreSQL ---- File Storage
        |
      Redis
        |
   RQ / Celery Workers
        |
 pandas / openpyxl / optional local LLM
```

### Web App

Принимает запрос, валидирует файл и шаблон, создаёт ProcessingJob и ставит его в очередь. Web-процесс не должен выполнять длительную конвертацию.

### PostgreSQL

Хранит пользователей, роли, версии шаблонов, ProcessingJob, статусы, quality_report, аудит и ссылки на файлы. Миграция с JSON выполняется через интерфейсы repositories.

### Redis и очередь

Redis/RQ подходит как простой следующий шаг для Team. Celery оправдан при сложной маршрутизации, расписаниях и нескольких типах workers. Очередь обеспечивает повторные попытки, отмену, heartbeat и управляемый параллелизм.

### Workers

Workers читают Excel из файлового хранилища, применяют сохранённый шаблон, формируют quality_report и сохраняют результат. Их можно масштабировать отдельно от web-приложения.

### File Storage

Для пилота достаточно локального volume. В production нужен корпоративный NAS, S3-совместимое хранилище или другой управляемый сервис с retention, правами доступа, шифрованием и резервным копированием.

## Роль локальной LLM

LLM опциональна и используется прежде всего для создания или уточнения правила. Массовая повторная обработка должна выполняться детерминированно по сохранённым и версионированным шаблонам. Это снижает стоимость, повышает воспроизводимость и упрощает аудит.

Пользовательские Excel-файлы не должны передаваться во внешние AI API. Локальная модель запускается в отдельном worker или сервисе только при наличии модели и согласованных ресурсов.

## Пример нагрузки Team

- 10–30 пользователей;
- сотни файлов в месяц;
- один web-инстанс;
- PostgreSQL;
- Redis/RQ;
- один или несколько workers;
- локальное или корпоративное файловое хранилище.

Параметры workers выбираются после измерения размеров файлов и времени обработки на пилоте.

## Пример нагрузки Enterprise

- несколько подразделений;
- тысячи файлов в месяц;
- несколько web-инстансов за reverse proxy;
- управляемый PostgreSQL;
- Redis и несколько очередей;
- специализированные worker pools;
- централизованное файловое хранилище;
- мониторинг, аудит, резервное копирование и интеграции.

Ориентир 30–100+ пользователей требует нагрузочного тестирования и не является гарантией текущего MVP.

## Следующий roadmap

1. PostgreSQL repositories и миграции.
2. Redis/RQ и асинхронные ProcessingJob.
3. Аутентификация, роли и versioned templates.
4. Retention и безопасная очистка файлов.
5. Метрики, structured logging и alerting.
6. Нагрузочное тестирование на профиле файлов заказчика.
7. Backup/restore и disaster recovery runbook.
